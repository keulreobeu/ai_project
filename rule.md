# 🚀 프로젝트 협업 가이드 (브랜치 및 커밋 컨벤션)

우리 팀의 원활한 협업과 깔끔한 코드 관리를 위한 브랜치 전략 및 커밋 규칙입니다. 
팀원 모두 본 규칙을 준수하여 개발을 진행해 주세요!

---

## 🌿 1. 브랜치 운용 전략

프로젝트는 **`main`**, **`develop`** 두 개의 중심 브랜치와 기능별 **`feature/`** 브랜치로 운영합니다.

* **`main`** : 제품으로 출시(배포)될 수 있는 가장 안정적인 코드가 담기는 브랜치입니다.
* **`develop`** : 다음 버전을 위해 개발 중인 코드가 모이는 중심 브랜치입니다.
* **`feature/기능이름`** : 각자 기능을 구현하는 작업 브랜치입니다.
    * *예시: `feature/login`, `feature/sidebar`*

### 💡 작업 흐름 (Workflow)
1. `develop` 브랜치에서 새로운 `feature/기능이름` 브랜치를 생성합니다.
2. 기능 구현 작업 후, 자신의 `feature` 브랜치에 커밋을 쌓습니다.
3. 작업이 완료되면 **`develop` 브랜치를 대상으로 Pull Request (PR)**를 생성합니다.
4. 팀원의 리뷰 및 승인을 거쳐 `develop` 브랜치에 병합(Merge)합니다.

---

## 💬 2. 커밋 메시지 컨벤션 (Commit Convention)

커밋 메시지는 코드의 변경 이유를 직관적으로 파악하기 위해 아래 규칙을 따릅니다.

### 📌 커밋 메시지 기본 구조
> **`태그: 커밋 내용`** 형태로 작성합니다. (태그 뒤 한 칸 띄우기)
> *예시: `feat: 구글 로그인 기능 구현`*

### 🏷️ 커밋 태그 종류

| 태그 (Type) | 설명 |
| :--- | :--- |
| **`feat`** | 새로운 기능에 대한 커밋 |
| **`fix`** | 버그 수정에 대한 커밋 |
| **`refactor`** | 코드 리팩토링에 대한 커밋 (기능 변경 없이 코드 구조만 개선) |
| **`design`** | 사용자 UI 디자인 변경에 대한 커밋 (CSS, 레이아웃 등) |
| **`docs`** | 문서 수정에 대한 커밋 (README, 위키 등) |
| **`test`** | 테스트 코드 수정 및 추가에 대한 커밋 |
| **`build`** | 빌드 관련 파일 수정에 대한 커밋 (의존성 추가, 설정 파일 변경 등) |
| **`chore`** | 그 외 패키지 매니저 설정 등 자잘한 수정에 대한 커밋 |
| **`merge`** | 풀 리퀘스트(PR) 합병에 대한 커밋 |

---

## 📝 3. 작성 예시 (Examples)

* `feat: 메인 페이지 상품 목록 조회 API 연동`
* `fix: 로그인 시 토큰이 만료되어도 자동 로그아웃되지 않는 오류 수정`
* `design: 네비게이션 바 컴포넌트 반응형 UI 적용`
* `docs: README.md 파일에 설치 및 실행 방법 추가`
